# 第 2 週今日練習題：使用者資料 CRUD API

## 題目說明

建立一個 Express 應用程式，連接 PostgreSQL 資料庫，實作完整的使用者資料 CRUD 操作。你將學會如何透過 `pg` 套件建立資料庫連線、執行 SQL 語句，並將結果以 JSON 格式回傳給客戶端。

---

## 技術要求

- **Express.js**：建立 RESTful API 端點，處理 HTTP 請求與回應
- **PostgreSQL**：使用 `pg`（node-postgres）套件連線，執行 INSERT、SELECT、UPDATE、DELETE 語句

---

## 具體任務

1. **初始化專案與資料庫**
   - 建立 `users` 資料表，欄位包含：`id`（SERIAL PRIMARY KEY）、`name`（VARCHAR(100)）、`email`（VARCHAR(150) UNIQUE）、`created_at`（TIMESTAMP DEFAULT NOW()）
   - 安裝必要套件：`express`、`pg`、`dotenv`
   - 建立 `.env` 檔案，儲存資料庫連線字串（`DATABASE_URL` 或個別欄位）

2. **建立資料庫連線模組**
   - 新增 `db.js`，使用 `pg.Pool` 建立連線池並匯出
   - 連線失敗時需在 console 顯示錯誤訊息

3. **實作以下 API 端點**
   - `GET /users` — 查詢所有使用者，依 `created_at` 降序排列
   - `GET /users/:id` — 依 ID 查詢單一使用者；找不到時回傳 404
   - `POST /users` — 新增使用者（body: `{ name, email }`），成功回傳 201 與新建立的資料
   - `PUT /users/:id` — 更新使用者 `name` 或 `email`；找不到時回傳 404
   - `DELETE /users/:id` — 刪除使用者；找不到時回傳 404，成功回傳 200 確認訊息

4. **測試所有端點**
   - 使用 `curl` 或 Postman 逐一測試每個端點
   - 確認新增重複 email 時，PostgreSQL 的 UNIQUE 限制會觸發錯誤，並在 API 回傳適當的錯誤訊息（422 或 409）

---

## 提示

- 使用 `pg.Pool` 而非 `pg.Client`，Pool 會自動管理連線，避免連線數耗盡
- SQL 語句使用參數化查詢（`$1, $2, ...`）防止 SQL Injection，例如：
  ```js
  pool.query('SELECT * FROM users WHERE id = $1', [req.params.id])
  ```
- `pool.query()` 回傳 Promise，記得使用 `async/await` 並加上 `try/catch`
- 查詢結果放在 `result.rows` 中，`result.rows[0]` 取第一筆
- 刪除或更新後，可用 `result.rowCount` 判斷是否有找到並影響到資料列

---

## 進階挑戰（選做）

- 將所有資料庫操作抽離至獨立的 `userService.js` 或 `userRepository.js` 模組，讓路由保持乾淨
- 新增 `PATCH /users/:id` 端點，只更新提供的欄位（部分更新）
- 為 `POST /users` 加上簡單驗證：`name` 不可空白、`email` 需符合基本格式，否則回傳 400
- 使用 `pg` 的 `pool.connect()` 手動取得 client，練習手動 `release()` 的時機
