# 第 3 週今日 Node.js 練習題：非同步任務排程器

## 題目說明

實作一個簡單的「非同步任務排程器」，模擬後端 API 中常見的非同步操作流程：依序讀取使用者資料、驗證權限、取得訂單清單。你需要分別用 **callback**、**Promise** 和 **async/await** 三種風格實作同一段邏輯，體驗各自的寫法差異與可讀性變化。

---

## 技術要求

- **概念**：callback、Promise chain、async/await、錯誤處理（try/catch vs .catch()）
- **需要的套件**：無（僅使用 Node.js 內建功能）

---

## 具體任務

1. **Callback 版本**：建立三個函式 `getUser`、`checkPermission`、`getOrders`，各自接受 callback 參數，使用 `setTimeout` 模擬非同步延遲（200ms）。在 callback 中依序呼叫，形成巢狀結構（callback hell），完成後印出最終結果。

2. **Promise 版本**：將上述三個函式改為回傳 Promise，使用 `.then().then().catch()` 串接，完成同樣的流程，並加上 `.catch()` 統一處理錯誤。

3. **async/await 版本**：用 `async function` 搭配 `await` 依序呼叫三個 Promise 函式，並用 `try/catch` 處理例外。

4. **錯誤情境**：在 `checkPermission` 中加入條件判斷——若使用者 ID 為 `999`，則回傳錯誤（callback 傳入 `Error`、Promise 使用 `reject`）。測試三種版本的錯誤處理是否正確觸發。

---

## 範例

```js
// --- 資料模擬 ---
const usersDB = {
  1: { id: 1, name: 'Alice', role: 'admin' },
  2: { id: 2, name: 'Bob', role: 'user' },
};

const ordersDB = {
  1: ['訂單A', '訂單B'],
  2: ['訂單C'],
};

// ===== Callback 版本 =====
function getUser_cb(userId, callback) {
  setTimeout(() => {
    const user = usersDB[userId];
    if (!user) return callback(new Error(`找不到使用者 ${userId}`));
    callback(null, user);
  }, 200);
}

function checkPermission_cb(user, callback) {
  setTimeout(() => {
    if (user.id === 999) return callback(new Error('權限不足'));
    callback(null, user);
  }, 200);
}

function getOrders_cb(user, callback) {
  setTimeout(() => {
    const orders = ordersDB[user.id] || [];
    callback(null, { user: user.name, orders });
  }, 200);
}

// 呼叫範例（callback hell）
getUser_cb(1, (err, user) => {
  if (err) return console.error('錯誤:', err.message);
  checkPermission_cb(user, (err, verifiedUser) => {
    if (err) return console.error('錯誤:', err.message);
    getOrders_cb(verifiedUser, (err, result) => {
      if (err) return console.error('錯誤:', err.message);
      console.log('[Callback] 結果:', result);
      // 輸出：[Callback] 結果: { user: 'Alice', orders: [ '訂單A', '訂單B' ] }
    });
  });
});

// ===== Promise 版本 =====
function getUser_p(userId) {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      const user = usersDB[userId];
      if (!user) reject(new Error(`找不到使用者 ${userId}`));
      else resolve(user);
    }, 200);
  });
}

function checkPermission_p(user) {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      if (user.id === 999) reject(new Error('權限不足'));
      else resolve(user);
    }, 200);
  });
}

function getOrders_p(user) {
  return new Promise((resolve) => {
    setTimeout(() => {
      resolve({ user: user.name, orders: ordersDB[user.id] || [] });
    }, 200);
  });
}

getUser_p(2)
  .then(checkPermission_p)
  .then(getOrders_p)
  .then(result => console.log('[Promise] 結果:', result))
  .catch(err => console.error('[Promise] 錯誤:', err.message));
// 輸出：[Promise] 結果: { user: 'Bob', orders: [ '訂單C' ] }

// ===== async/await 版本 =====
async function fetchUserOrders(userId) {
  try {
    const user = await getUser_p(userId);
    const verified = await checkPermission_p(user);
    const result = await getOrders_p(verified);
    console.log('[async/await] 結果:', result);
  } catch (err) {
    console.error('[async/await] 錯誤:', err.message);
  }
}

fetchUserOrders(1);
// 輸出：[async/await] 結果: { user: 'Alice', orders: [ '訂單A', '訂單B' ] }

fetchUserOrders(999);
// 輸出：[async/await] 錯誤: 找不到使用者 999
```

---

## 提示

- `setTimeout(fn, 200)` 可模擬資料庫查詢或 API 呼叫的延遲。
- Callback 的錯誤慣例是 **Error-first**：`callback(err, data)`，成功時 `err` 為 `null`。
- Promise 的 `.catch()` 可以攔截整條 chain 中任何一個 `.then()` 拋出的錯誤。
- `async/await` 讓非同步程式碼讀起來像同步，但底層仍是 Promise。
- 在 `async` 函式中，若不加 `try/catch`，未處理的 rejection 會造成 `UnhandledPromiseRejection`。

---

## 學習重點

這三種模式在後端 API 開發中無所不在：

- **Callback**：Node.js 早期 API（如 `fs.readFile`）使用此風格；理解它有助於閱讀舊有程式碼。
- **Promise**：Express 路由中常見，適合需要明確 chain 的場景（如先驗證再查 DB）。
- **async/await**：現代 Express / Fastify / NestJS 的主流寫法，可讀性最高，與 `try/catch` 搭配錯誤處理最直覺。

掌握這三種風格後，你就能讀懂並維護任何 Node.js 後端專案的非同步邏輯。

---

## 進階挑戰（選做）

1. **並行處理**：若使用者有多筆訂單 ID，用 `Promise.all()` 同時取得所有訂單詳情，而非逐一等待。
2. **逾時機制**：為每個 Promise 加上逾時保護（若超過 500ms 未回應則 reject），可用 `Promise.race()` 實作。
3. **重試邏輯**：若 `getOrders` 失敗，自動重試最多 3 次，每次間隔 300ms。
