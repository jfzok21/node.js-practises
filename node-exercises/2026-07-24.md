# 第 3 週今日 Node.js 練習題：非同步資料讀取流程控制

## 題目說明

你需要模擬一個簡單的「使用者資料載入系統」，依序從三個非同步步驟取得資料：
1. 讀取使用者 ID 清單
2. 根據 ID 取得使用者詳細資料
3. 根據使用者資料取得其訂單記錄

這個練習將分三個版本實作：**Callback**、**Promise**、**async/await**，讓你深刻感受三種寫法的差異。

## 技術要求

- 概念：Callback Hell、Promise Chain、async/await、錯誤處理
- 需要的套件：無（純 Node.js）

## 具體任務

### 版本一：Callback（理解地獄的起點）

1. 建立 `callback-version.js`
2. 實作 `getUserIds(callback)`，模擬非同步（用 `setTimeout` 延遲 500ms）回傳 `[1, 2, 3]`
3. 實作 `getUserById(id, callback)`，延遲 300ms 回傳 `{ id, name: 'User_' + id }`
4. 實作 `getOrdersByUserId(userId, callback)`，延遲 200ms 回傳 `[{ orderId: userId * 100 }]`
5. 巢狀呼叫三個函式，取得第一個使用者的訂單

### 版本二：Promise（鏈式寫法）

1. 建立 `promise-version.js`
2. 將上述三個函式改為回傳 `Promise`
3. 使用 `.then().then().then()` 鏈式呼叫
4. 在最後加上 `.catch()` 處理錯誤
5. 在 `getUserById` 中，若 id 為 `0` 則 `reject` 一個錯誤，測試錯誤捕捉

### 版本三：async/await（現代寫法）

1. 建立 `async-version.js`
2. 沿用版本二的 Promise 函式
3. 用 `async function main()` 包裹，使用 `await` 依序呼叫
4. 用 `try/catch` 包裹整個流程，處理錯誤
5. 使用 `Promise.all()` 同時取得所有使用者（id 1, 2, 3）的詳細資料，比較與逐一等待的效率差異

## 範例

```js
// async-version.js 的預期輸出
// 逐一取得：
User: { id: 1, name: 'User_1' }
Orders: [ { orderId: 100 } ]

// Promise.all 同時取得所有使用者：
All users: [
  { id: 1, name: 'User_1' },
  { id: 2, name: 'User_2' },
  { id: 3, name: 'User_3' }
]

// 錯誤測試（id 傳入 0）：
Error caught: Invalid user id: 0
```

## 提示

- `setTimeout(fn, ms)` 可模擬非同步 I/O 延遲
- Callback 的錯誤慣例是 `callback(error, data)`，第一個參數為 error
- `new Promise((resolve, reject) => { ... })` 是把非同步包成 Promise 的標準寫法
- `Promise.all([p1, p2, p3])` 會等所有 Promise 都完成才繼續
- `async` 函式一定回傳 Promise，即使你沒有明確 `return`
- `await` 只能在 `async` 函式內使用

## 學習重點（跟後端 API 的關聯）

在真實的 Express API 中，**幾乎所有操作都是非同步的**：
- 資料庫查詢（`await db.query()`）
- 呼叫外部 API（`await fetch()`）
- 讀取檔案（`await fs.promises.readFile()`）

掌握 async/await 是寫出乾淨 Express Route Handler 的前提。例如：

```js
// Express route 的典型寫法
app.get('/users/:id/orders', async (req, res) => {
  try {
    const user = await getUserById(req.params.id)
    const orders = await getOrdersByUserId(user.id)
    res.json({ user, orders })
  } catch (err) {
    res.status(500).json({ error: err.message })
  }
})
```

2. **錯誤重試機制**：實作 `withRetry(asyncFn, maxRetries)` 高階函式，在失敗時最多重試 N 次
